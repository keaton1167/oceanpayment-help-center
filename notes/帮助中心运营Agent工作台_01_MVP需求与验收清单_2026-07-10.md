# 帮助中心运营 Agent 工作台 MVP 需求与验收清单

更新日期：2026-07-10  
关联项目：`D:\AI_Workspace\oceanpayment-help-center`

当前状态：开发前需求冻结草案

## 1. MVP 目标

第一版目标不是做一个“大而全后台”，而是让运营人员可以不直接接触 Docusaurus 源码，也能完成帮助中心内容维护闭环：

```text
飞书登录
  -> 初始化当前帮助中心文档
  -> 新增/编辑/导入文档
  -> 转换为符合当前样式的 MDX
  -> 人工审核
  -> 构建检查
  -> 生成 build.zip
  -> 飞书通知技术支持部手动部署
```

## 2. MVP 成功标准

MVP 完成时，应满足以下状态：

- 运营人员可以通过飞书登录进入系统。
- 系统能从当前 Docusaurus 项目读取已上线文档、目录、附件和图片。
- 运营人员可以新建、编辑、保存草稿、提交审核文章。
- 运营人员可以从飞书文档导入单篇文章，并生成 MDX 草稿。
- 系统可以生成中文和英文文档之间的关联记录。
- 审核人可以通过或驳回文档。
- 管理员可以触发构建检查并生成 `build.zip`。
- 系统可以生成本次变更清单和飞书交接通知。
- build 包仍由技术支持部同事手动部署，不由系统直接部署生产服务器。

## 3. 角色范围

| 角色 | 第一版职责 | 权限边界 |
| --- | --- | --- |
| 运营人员 | 新增、编辑、导入、同步、提交审核文档 | 不能直接生成正式 build 包，不能跳过审核 |
| 审核人员 | 审核文档内容和英文草稿 | 可以通过或驳回，不能直接部署 |
| 管理员 | 管理目录、触发构建、生成 build 包、通知技术支持部 | 不能绕过系统记录直接标记上线 |
| 技术支持部 | 接收 build 包并手动部署公司服务器 | 第一版不需要登录系统，可通过飞书消息接收信息 |

## 4. P0 功能清单

### 4.1 登录与权限

需求：

- 支持飞书登录。
- 识别用户姓名、邮箱、open_id、部门信息。
- 支持基础角色：运营人员、审核人员、管理员。
- 未授权用户不能进入系统。

验收：

- 未登录访问后台会跳转登录。
- 登录后能看到用户身份。
- 不同角色看到的操作按钮不同。
- App Secret 不在前端暴露。

### 4.2 当前项目初始化

需求：

- 扫描 `docs` 中文文档。
- 扫描 `i18n/en/docusaurus-plugin-content-docs/current` 英文文档。
- 读取 `_category_.json`。
- 读取 MDX front matter。
- 识别图片和附件引用。
- 生成文档、目录、资源、双语关联初始数据。

验收：

- 初始化后的文档数量与当前实际上线页面基本对齐。
- 每篇文档有标题、语言、目录、slug、正式 URL、源文件路径。
- 中英文文档不会错位混在同一条记录中。
- 缺失资源会被标记为异常，而不是静默忽略。

### 4.3 内容管理

需求：

- 文档列表支持筛选、搜索、分页。
- 支持按语言、目录、状态、负责人、来源筛选。
- 支持新建文档。
- 支持编辑文档标题、摘要、目录、slug、正文、附件、负责人。
- 支持保存草稿。
- 支持提交审核。
- 支持查看文档详情和历史版本。

验收：

- 新建文档不会直接进入待发布状态。
- slug 冲突时系统阻止保存。
- 修改目录时能提示正式 URL 可能变化。
- 正文中引用的图片和附件能被系统识别。

### 4.4 目录管理

需求：

- 支持一级目录和二级目录维护。
- 支持中英文目录名。
- 支持目录排序。
- 支持启用/停用目录。
- 支持生成或更新 `_category_.json`。

验收：

- 有文档使用的目录不能直接删除。
- 修改目录 slug 前有影响范围提示。
- 目录排序能映射到 Docusaurus sidebar 展示。

### 4.5 飞书单篇文档导入

需求：

- 支持粘贴飞书文档链接。
- 系统解析 token。
- 读取飞书文档标题和正文块。
- 下载图片和附件。
- 转换为 Markdown/MDX 草稿。
- 生成目录、slug、摘要建议。
- 创建飞书映射关系。

验收：

- 普通标题、段落、列表、表格、链接可以正常转换。
- 图片和附件下载失败时有明确错误。
- 导入不会直接覆盖已上线文档。
- 导入后可人工编辑再提交审核。

### 4.6 MDX 转换与样式适配

需求：

- 生成符合当前 Docusaurus 项目的 MDX。
- 生成 front matter。
- 图片路径写入 `/img/help-center/...`。
- 附件路径写入 `/files/help-center/...`。
- 保持当前附件预览逻辑兼容。
- 避免生成错误资源前缀。

验收：

- 生成的 MDX 可以通过 Docusaurus build。
- 文档样式与当前帮助中心一致。
- 附件链接能触发当前附件预览或下载逻辑。
- 中文和英文路径分别写入正确目录。

### 4.7 英文草稿生成

需求：

- 支持从中文文档生成英文草稿。
- 保留 MDX 结构、链接、图片和附件。
- 标记为“英文待复核”。
- 人工确认后才能进入待发布。

验收：

- AI 生成英文不会直接标记为已审核。
- 英文正文不会丢失附件和图片。
- 中英文文档有明确关联关系。

### 4.8 审核流程

需求：

- 草稿可以提交审核。
- 审核人可以通过或驳回。
- 驳回必须填写原因。
- 审核动作写入日志。

验收：

- 未审核文档不能进入正式 build 范围。
- 驳回后运营可以继续修改。
- 审核记录可以追溯到人和时间。

### 4.9 构建检查与 build 包

需求：

- 管理员可以触发构建检查。
- 构建前执行资源检查。
- 执行 `npm run build`。
- 生成 `build.zip`。
- 生成变更清单。
- 生成部署交接说明。

验收：

- 构建失败时展示可读错误。
- 资源缺失时阻止生成正式包。
- build 包不包含管理后台代码。
- build 包结构与当前 Docusaurus 部署包一致。

### 4.10 飞书通知技术支持部

需求：

- build 包生成后可发送飞书群通知。
- 通知包含变更摘要、下载链接、部署说明、验收链接。
- 如 build 包过大，可发送飞书云盘或系统下载链接。

验收：

- 通知内容不包含 App Secret、Token 等敏感信息。
- 技术支持部能通过消息完成下载和部署理解。

## 5. P1 后续功能

- 飞书知识库批量导入。
- 已导入文档手动同步。
- 批量同步任务。
- 内容质检 Agent。
- 英文版本差异检测。
- 附件预览兼容性检查。
- 内部链接有效性检查。
- build 包历史记录。
- 技术支持部部署确认入口。

## 6. P2 后续增强

- 飞书机器人自然语言指令。
- 定时同步。
- 版本 diff 与回滚辅助。
- 按目录或业务线分配审核人。
- Git PR/CI 对接。
- 自动生成上线周报。

## 7. 第一版不做

- 不直接部署公司服务器。
- 不自动跳过审核。
- 不让 AI 内容未经复核直接上线。
- 不把管理系统打进公开帮助中心 build 包。
- 不迁移旧 `miaoda-app` 数据。
- 不在前端页面明文管理密钥。
