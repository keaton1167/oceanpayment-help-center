# 帮助中心管理系统与 Agent 方案设计

更新时间：2026-07-10  
项目路径：`D:\AI_Workspace\oceanpayment-help-center`  
参考系统：`D:\AI_Workspace\oceanpayment-help-center\miaoda-app`  
当前定位：方案讨论基线，暂不进入开发实现

## 1. 背景

当前 Oceanpayment 帮助中心已经由 Docusaurus 搭建完成，文档内容、双语结构、来源限制、搜索、附件预览和主维护清单已完成本轮整理。现阶段的主要问题已经从“站点能否上线”转向“后续如何持续维护”。

目前帮助中心仍主要依赖人工维护，包括新增文档、修改文档、补充英文版、处理附件、执行构建、生成部署包、给技术支持部同事交付 build 包。为了让运营部门可以自主维护帮助中心，需要建设一个面向运营人员的管理系统，并在系统中引入 Agent 能力，用来自动化处理飞书导入、格式检查、英文生成、构建交付和飞书通知。

此前在飞书妙搭中已经搭建过“帮助中心管理系统”，代码已放在 `miaoda-app` 目录。该系统可以作为业务流程和页面能力参考，但不建议原样迁移，因为它强依赖飞书妙搭和 Lark APaaS 能力，包括认证、数据库、文件服务、用户组件、权限体系和发布环境配置。

本阶段目标是重新规划一个不强依赖妙搭的帮助中心管理方案，同时继续兼容飞书生态协作。

## 2. 总体定位

建议采用“独立应用系统 + Agent 自动化能力”的组合形态。

应用系统负责稳定、可控、可审计的业务操作，包括文档列表、内容编辑、目录设置、附件上传、飞书导入、审核流、构建包生成、任务日志和权限控制。

Agent 负责提升运营效率，包括飞书文档转换、知识库批量导入、格式检查、资源异常分析、英文版生成、变更清单生成、部署交接说明生成和飞书消息通知。

该方案既可以满足“运营人员有系统可操作”的现实需要，也可以支撑“Agent 应用落地推进”的 OKR。系统不是纯聊天式 Agent，而是一个带 Agent 能力的运营工作台。

## 3. 关键原则

- 不再强依赖飞书妙搭。飞书仍作为登录、文档源、知识库源、通知渠道和协作平台。
- 不让运营直接接触 Docusaurus 源码目录。运营通过系统表单和编辑器维护内容。
- 不由系统直接部署公司服务器。系统生成 build 包，由技术支持部同事手动部署。
- 所有进入 build 包的内容必须有状态、记录、校验日志和责任人。
- 当前 Docusaurus 项目是最终内容源之一。系统需要能从当前项目重新初始化文档数据。
- 旧 `miaoda-app` 不迁移数据，只参考功能和交互。

## 4. 角色与权限

### 4.1 运营人员

所属部门：客户关系部。  
主要职责：新增文档、修改文档、上传附件、导入飞书文档、发起同步、生成英文草稿、提交审核、查看预览、跟踪上线状态。

建议权限：

- 查看文档列表和文档详情
- 新建和编辑草稿
- 上传图片和附件
- 从飞书文档或知识库导入
- 手动同步已建立映射的飞书文档
- 发起英文版生成
- 提交审核
- 查看构建结果和异常说明

### 4.2 审核人员

所属部门：客户关系部。  
主要职责：审核运营提交的文档变更，确认内容是否可以进入待打包状态。

建议权限：

- 查看待审核文档
- 审核通过
- 驳回并填写原因
- 查看变更对比
- 查看 AI 检查建议和资源检查结果

### 4.3 管理员

主要职责：系统配置、目录维护、最终生成 build 包、触发飞书通知、处理异常。

建议权限：

- 管理一级/二级目录
- 管理用户权限
- 触发全站构建检查
- 生成正式 build 包
- 下载 build 包
- 发送飞书群交接通知
- 查看完整任务日志
- 标记已交付和已上线

### 4.4 技术支持部同事

主要职责：接收 build 包并手动部署到公司服务器。

建议交付方式：

- 飞书群消息通知
- 消息中包含 build 包下载入口
- 消息中包含本次变更清单
- 消息中包含部署说明和验收链接
- 如文件较大，可改为飞书云盘链接或系统下载链接

## 5. 完整闭环流程

### 5.1 手动新增文档流程

1. 运营登录系统。
2. 点击新建文档。
3. 填写标题、摘要、语言、一级目录、二级目录、slug、正文、负责人。
4. 上传图片或附件，并插入正文。
5. 系统自动生成帮助中心路径预览。
6. 系统执行格式检查和资源检查。
7. 运营保存草稿。
8. 运营提交审核。
9. 审核人通过后，文档进入待发布。
10. 管理员生成 build 包。
11. 系统生成部署交接说明并通知飞书群。
12. 技术支持部部署。
13. 运营或管理员标记已上线。

### 5.2 飞书文档导入流程

1. 运营粘贴飞书文档链接。
2. 系统解析文档 token，读取标题和正文。
3. 系统下载图片和附件。
4. Agent 将飞书内容转换为适合 Docusaurus 的 MDX。
5. 系统生成标题、摘要、slug 和目录建议。
6. 运营确认映射关系。
7. 系统创建帮助中心文档草稿。
8. 后续进入审核、构建、交付流程。

### 5.3 飞书知识库批量导入流程

1. 运营选择或粘贴飞书知识库链接。
2. 系统读取知识库空间和目录树。
3. 运营选择要导入的节点。
4. 系统批量生成映射草稿。
5. Agent 给出目录映射、slug、语言和标题建议。
6. 运营批量确认。
7. 系统批量创建文档或同步映射。
8. 同步后生成任务日志和异常清单。

### 5.4 同步流程

1. 运营或管理员在导入工具中查看飞书映射列表。
2. 选择单篇或批量同步。
3. 系统重新读取飞书文档内容。
4. 系统转换 MDX 并更新帮助中心文档草稿。
5. Agent 检查标题层级、链接、附件、表格、图片路径。
6. 同步成功则记录版本。
7. 同步失败则给出可读错误原因和修复建议。

### 5.5 Build 包交付流程

1. 管理员在发布中心选择生成正式 build 包。
2. 系统写入待发布文档到 Docusaurus 项目结构。
3. 系统执行 Docusaurus build。
4. 系统执行资源完整性检查。
5. 系统生成 build.zip。
6. 系统生成文档清单、变更摘要、部署说明。
7. 系统发送飞书群通知给技术支持部同事。
8. 技术支持部下载 build 包并手动部署。
9. 部署完成后，运营或管理员标记上线完成。

## 7. Agent 能力设计

Agent 不应替代所有页面操作，而应嵌入关键流程，作为运营助手和自动化执行层。

### 7.1 内容转换 Agent

能力：

- 从飞书 Docx 转换为 MDX
- 保留标题、列表、表格、链接、图片、附件
- 自动修正常见格式问题
- 生成适合帮助中心的文档摘要
- 生成 slug 建议
- 生成目录建议

### 7.2 内容质检 Agent

能力：

- 检查标题层级
- 检查 H1/H2/H3 是否符合帮助中心规范
- 检查内部链接
- 检查外部链接
- 检查图片路径
- 检查附件路径
- 检查 0 字节附件
- 检查正文是否包含飞书导出残留文本
- 检查中英文文档是否错位
- 检查是否存在 Docusaurus build 风险

输出：

- 通过
- 警告
- 阻塞
- 修复建议

### 7.3 翻译 Agent

能力：

- 根据中文生成英文草稿
- 根据中文更新英文版
- 对比中英文更新时间，标记英文待更新
- 保留 MDX 结构和附件链接
- 保留专业术语
- 输出待人工复核的英文文档

安全边界：

- 第一阶段 AI 只生成草稿
- 英文版必须由运营或审核人确认后才能进入待发布

### 7.4 发布交接 Agent

能力：

- 生成本次 build 包变更摘要
- 生成技术支持部部署说明
- 汇总新增、修改、归档文档
- 汇总附件和资源异常
- 生成飞书群通知文案
- 生成上线验收 checklist

### 7.5 飞书机器人 Agent

后续可扩展为飞书机器人，支持在飞书中触发部分动作。

示例指令：

- “同步这篇飞书文档到帮助中心”
- “检查这篇文档是否可以发布”
- “生成本次 build 包说明”
- “查看今天同步失败的文档”
- “提醒审核人处理待审核文档”

## 10. 推荐技术路线

### 10.1 前端

建议使用独立 React 管理后台。

可选方案：

- React + Vite
- React Router
- TanStack Query
- shadcn/radix 组件
- Tiptap 或 Markdown 编辑器

前端设计方向：

- 运营工具型界面
- 信息密度适中
- 保持旧系统的左侧导航和表格工作流
- 减少营销式页面
- 强化任务状态、异常提示和操作反馈

### 10.2 后端

建议使用 Node.js 服务。

可选方案：

- NestJS：更适合模块化和后续扩展
- Fastify/Express：更轻量，适合快速 MVP

考虑到旧系统已经是 NestJS，业务模块可以参考旧结构，建议新系统仍采用 NestJS 或至少保持模块化边界。

### 10.3 数据库

建议正式环境使用 PostgreSQL。

MVP 阶段可选：

- SQLite：本地快速验证
- PostgreSQL：更贴近正式环境

建议数据表：

- users
- roles
- documents
- categories
- document_versions
- resources
- feishu_mappings
- sync_tasks
- build_tasks
- review_tasks
- notifications
- system_configs

### 10.4 文件与构建

文件落地目标：

- 文档写入 Docusaurus `docs` 或 `i18n/en/...`
- 图片写入 `static/img/help-center/...`
- 附件写入 `static/files/help-center/...`
- 构建输出 `build`
- 交付包输出 zip

构建动作：

- `npm run build`
- 文档审计脚本
- 资源完整性检查
- 搜索索引检查
- 附件预览资源检查

### 10.5 部署

部署位置当前未定，建议分阶段处理。

阶段一：

- 本地或内部服务器试运行
- 先打通数据初始化、文档编辑、飞书导入、build 包生成

阶段二：

- 部署到公司内网或公司服务器
- 配置飞书登录回调
- 配置飞书机器人事件
- 配置 build 包下载地址

阶段三：

- 接入更正式的权限、审计、备份和监控

## 15. 风险与注意事项

- 飞书文档和知识库权限需要提前确认，尤其是图片和附件下载权限。
- App Secret 不应由运营在页面维护。
- 构建包可能较大，飞书群直接发送可能遇到大小限制。
- 技术支持部手动部署后，需要有“已部署确认”入口，否则系统无法知道真实上线状态。
- AI 生成英文内容必须人工复核。
- 直接从飞书同步可能覆盖运营在系统内做的手动修改，需要设计冲突提示。
- 当前帮助中心已有特殊英文页面规则，初始化时不能只按物理文件夹判断语言。
- Docusaurus 构建失败信息需要转成运营可读的错误说明。

## 16. 需要后续确认的问题

- 管理系统最终部署位置。
- build 包发送方式是否长期采用飞书群直接发送，还是改为飞书云盘或系统下载链接。
- 技术支持部部署完成后由谁在系统标记“已上线”。
- 审核人员是固定人员，还是按目录/业务线分配。
- 飞书机器人是否需要第一版上线。
- Agent 使用的大模型和调用方式。
- 是否需要保存每次文档版本差异。
- 是否需要把构建包和文档清单长期归档。

## 18. 当前推荐结论

建议将下一阶段项目正式定义为：

> 帮助中心运营 Agent 工作台

它不是单纯后台，也不是纯聊天机器人，而是一个以运营后台为主体、以 Agent 自动化为增强能力的应用。

第一版先实现完整闭环，但把高风险自动化控制住：

- AI 可以生成和检查，但不能绕过人工审核。
- 系统可以生成 build 包，但不直接部署公司服务器。
- 飞书可以导入和通知，但不作为唯一数据源。
- 当前 Docusaurus 项目仍然是最终站点生成基础。
